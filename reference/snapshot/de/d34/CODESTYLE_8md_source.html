3 This document explains the codestyle that the Aurora Framework uses. We use this rules to keep our code spaghetti safe and with a better look.

5 ## Aurora Codestyle Specification
6 This specification is based on K&R variants: 1TBS, Stroustrup, Linux kernel and BSD KNF codestyles.

32 ### Comments
33 In Aurora code files, we have a very well defined anatomy for comment blocks. You have three types of comments: the initial comments, documentation comments and code comments. All of them need to be written in English and well explained for easier development of other programmers that want to contribute to the project.
34 
35 #### Initial Comments
36 Initial comments have general information about the file and should be present on ALL source code files. We have a boilerplate for C-like languages and languages that have `#` as 'keyword' for comment blocks. Depending on the license of the repository, you should use GPL-3.0 and LGPL-3.0 boilerplates. You can find them [here](BOILERPLATES.md).
37 
38 
39 #### Documentation Comments
40 To compile our documentation we use Doxygen, which uses JavaDoc Style for C-like languages and Doxygen style for other languages as a comment standard for automatically code structure detection. When you write a code file you should use this structure on header files:
41 ```cpp
42 /** Example comment
43  * Another comment
44  *
45  */
46 ```
47 or this, in case of a non C-like language:
48 ```cmake
49 ## Example comment
50 # Another comment
51 #
52 ```
53 
54 You should use special commands so Doxygen knows what you want to do with the comment. Here are some useful and most used commands:
55 - `@file`
56 - `@param`
57 - `@return`
58 - `@see`
59 - `@author`
60 
61 You can find the full list [here](http://www.stack.nl/~dimitri/doxygen/manual/commands.html).
62 
63 For better codestyle we activated the `JAVADOC_AUTOBRIEF`, which means that you can automatically initialize a brief description ending with a first dot followed by a space or a new line. For example:
64 ```cpp
65 /** Brief description which ends at this dot. Details follow
66  * here.
67  */
68 ```
69 **Note:** Because of the aesthetics, use the brief in the first line (follow the example) whenever possible.
70 
71 Each file must have a brief description, a `@file` tag, which has a partial path of the file and a `@since` tag to determinate when this file was created (framework version). It can also have the general author and contributers of the file, using `@author` or `@authors`.
72 
73 `@file` tag must have this syntax: `module-name/src/File.cpp` or `AuroraFW/Module/Header.h`.
74 
75 Each function must have an initial comment which explains:
76 - What its purpose or use is.
77 - What arguments it requires or accepts and what their use is.
78 - What value(s) the function returns.
79 - The version number of the framework since the first implementation.
80 
81 It can also have the creator of the function, but its not mandatory.
82 
83 In these comments you should use a formal and clear language, because the purpose of those comments are to automatically create the documentation reference of the framework.
84 
85 Here is a practical example of a documentation comment:
86 ```cpp
87 /** Constructs a vector with the given coordinates.
88  * @param x The x value for the x coordinate.
89  * @param y The y value for the y coordinate.
90  * @see Vector2D( )
91  * @see Vector2D(float )
92  */
93 ```
94 You can see the result [here](https://aurora-fw.github.io/docs/reference/snapshot/d7/dc9/structAuroraFW_1_1Math_1_1Vector2D.html#aa1573e1f3c75fae2b0e8ebcbd3a59adc).
95 
96 You can also use single comments for documentation: `///`.
97 
98 #### Code Comments
99 These types of comments can be used on headers and source files. The purpose of these comments is to help people understand an instruction or even a block of code. We use the common syntax of a comment: `//` and:
100 ```cpp
101 /*
102 block comments
103 */
104 ```
105 Here you just need to be clear on what you want to explain to the programmer, so no formal language is needed and, for better communication, use easy words.

107 ### Naming
108 To name our code we use typical methods adopted in programming to know what's the type of the instructions used. We use a different name idention for types, functions and objects/variables.
109 
110 #### Types
111 For types such as a `class`, `struct`, `typedef`, `enum`, `namespace`, etc, you should use PascalCase, which means that the first letter of each concatenated word is capitalized. Here are some examples:
112 
113 - `BackColor`
114 - `TimeUtc`
115 - `Timer`
116 
117 #### Functions
118 For functions you should use camelCase, which means that the first word is in lowercase and the rest of the words start with a capital letter. Here are some examples:
119 - `getName()`
120 - `setName()`
121 - `isNull()`
122 
123 #### Objects and Variables
124 For objects or variables we write them in lowercase. Specifically, for private members use an `_` at the begin.
125 
126 **Note**: Words that are already assigned by the language (keywords) should have an `_` at the end.
127 
128 #### Acronyms
129 If the first letter is uppercase then the whole acronym should have uppercase letters. Else if the first letter is lowercase then the whole acronym should have lowercase letters. Here are some examples:
130 - `ASCIIArt`
131 - `asciiArt`

133 ### Lines
134 Line structure is important for programmers. If lines are separated and with a length limit, code readability is improved and the development workflow is way better. It's important that blank lines exist and may be added to separate different blocks of code. It doesn't affect the compiler in any way, so there's no excuse for not using them.
135 
136 #### Length
137 When talking about soft limit, the lines should not pass 80 characters. For hard limit, lines must not pass 120 characters.
138 
139 #### Ending
140 For line ending we use Unix LF (linefeed). If you are developing on Windows, you should use a linefeed compatible editor.
141 
142 ### File encoding
143 You must use 8-bit unicode, UTF-8.
144 
145 ### Braces
146 If it's a function, don't open braces in the same line as the declarations, else please open it in the same line. To close braces you must always do it in a new line, unless its has no body.
147 
148 #### Single and multiple statements
149 Do not unnecessarily use braces where a single statement will do.
150 ```cpp
151 if (condition)
152  action();
153 ```
154 and
155 ```cpp
156 if (condition)
157  do_this();
158 else
159  do_that();
160 ```
161 
162 This does not apply if only one branch of a conditional statement is a single statement; in the latter case use braces in both branches:
163 ```cpp
164 if (condition) {
165  do_this();
166  do_that();
167 } else {
168  otherwise();
169 }
170 ```

172 ### Tab idention
173 Use tab instead of spaces for tab idention, if supported in the language, and configure your editor for 4 spaces in a single tab. Then, for alignment, use spaces. This helps to reduce significantly the project size.
174 
175 ### Spaces
176 Use a space after these keywords: `if`, `switch`, `case`, `for`, `do`, `while`, but not with `sizeof`, `typeof`, `alignof`, or `__attribute__`. Also don't add spaces around (inside) parenthesized expressions.
177 
178 #### Pointers
179 When declaring pointer data or a function that returns a pointer type, the preferred use of * is adjacent to the data name or function name and not adjacent to the type name. For example:
180 ```cpp
181 char *name;
182 unsigned int memory(char *ptr, char **retptr);
183 char *convert(string *s);
184 ```
185 
186 ### Headers
187 In header files, you should write functions without argument names, or if needed, use very short names, like `s`, `t`, `str`, `n`, `i`, etc.
188 Always use the global header on ALL headers (`AuroraFW/Global.h`). Use aurora defined data types, like `ArInt_t`, `ArInt8_t`, etc.
189 
190 ### Macros
191 Names of macros defining constants and labels in enums are capitalized.
192 Enums are preferred when defining several related constants.
193 CAPITALIZED macro names are appreciated but macros resembling functions may be named in lower case.
194 Generally, inline functions are preferable to macros resembling functions.
195 
196 Macros with multiple statements should be enclosed in a do - while block:
197 ```cpp
198 #define macrofun(a, b, c) \
199  do { \
200  if (a == 5) \
201  do_this(b, c); \
202  } while (0)
203 ```
204 
205 Global macros must be defined on headers and shouldn't be used on source files (only if strictly needed).
